Here's the full picture:

**Crates / Project Structure**
- `zk-pki-primitives` — shared types, traits, SCALE helpers, no runtime deps, `no_std` compatible
- `zk-pki-tpm` — TPM attestation parsing, EK extraction, trust chain verification, depends only on primitives
- `zk-pki-pallet` — all FRAME extrinsics, storage, `on_initialize`, lookup table, EK registry, TTL, purge, atomic ops
- `zk-pki-rpc` — JSON-RPC node extension, off-chain query layer, depends on primitives and pallet via runtime-api
- `zk-pki-runtime` — reference runtime wiring the pallet in, for integration testing
- `zk-pki-client` — mobile app Rust layer, TPM extrinsic construction, TOTP handoff, contract signing
- Dependency graph is strictly one-directional, nothing depends on `rpc` or `runtime`, `primitives` depends on nothing internal

**Smart Contracts (ink!)**
- EK trust chain registry contract — known-good TPM vendor cert chains, multisig maintained
- Compromised chain advisory contract — append-only, write-once entries with block timestamp, higher signature threshold than registry, time-lock or challenge period before entries take full effect
- Both contracts emit events consumed by reputation scoring via off-chain worker
- Both use multisig with separation of duties — not just M-of-N but representative parties must sign, no single org can unilaterally write
- Roles represented: TPM vendors, independent security auditors, ZK-PKI governance body
- Registry stores EK root cert hash → vendor name, firmware version range, trust tier, date added, added by whom

**Reputation Scoring Requirements**
- Queries both advisory contracts when evaluating any cert or issuer
- Positive weight: EK chain present in known-good registry
- Heavy negative weight: any part of EK chain in compromised advisory
- `mint_block` vs compromised entry timestamp determines retroactive vs prospective treatment — preserves non-repudiation
- New roots start with low reputation, climb slowly with age, volume of valid certs, and low invalidation rate
- Roots appearing shortly after a related compromise event start in negative territory
- On-chain address lineage tracking — proxy account history, funding sources, association with previously compromised operators
- Relying parties set their own reputation thresholds
- RPC API exposes raw reputation score plus contributing factors, recommended floor is advisory not enforced

**Issuer Compromise Rules**
- A compromised issuer's entire issuance history is considered suspect — no surgical cert rescue
- `compromised_at_block` is write-once on issuer records, set by root
- If compromised issuer principals spin up a new root, reputation scoring detects address lineage and starts that root in negative territory
- Relying parties seeing a tainted issuer history should find a new issuer — the protocol does not paper over it

**Root Compromise Rules**
- Requires dedicated fast-track governance mechanism for emergency flagging — standard referendum too slow for active compromise
- Full referendum ratifies after the fact
- `compromised_at_block` is write-once, OpenGov controlled, cannot be overwritten
- Certs minted before `compromised_at_block` remain in lookup table, relying parties compare `mint_block`
- Certs minted on or after `compromised_at_block` automatically invalidated by pallet

**TPM / Attestation Requirements**
- Physical TPM or secure enclave mandatory for PoP eligibility
- Full EK certificate chain validation against known-good registry, not just format checking
- EK deduplication enforced — one active PoP cert per registered EK
- Forged or unrecognized EK chains surface as reputation signal, not hard rejection alone
- TOTP secret generated atomically with keypair at genesis, transferred to local authenticator app, user operational responsibility thereafter
- Private key never leaves device — cloud-synced authenticator use is user's operational risk, not a protocol flaw
- Packed attestation valid for machine/org identity only, not PoP eligible
- Mobile app interfaces locally between user and TPM calls

**Certificate Rules**
- Certs are monolithic — no key rotation, no field updates
- Thumbprint computed on-chain at mint via Blake2b-256 over SCALE-encoded canonical fields
- `schema_version` is first field in canonical serialization for migration detection
- NFT belongs to user — issuer can invalidate lookup presence but cannot burn the token
- Both table presence AND `is_active = true` required for a cert to be considered valid — belt and suspenders, no exceptions
- 30-day grace period after expiry is the holder's self-discard window, not a validity window
- Reissuance is atomic — old entry deleted, new entry written, EK registry updated, all in same execution
- Genesis contract offers include a unique block-bound nonce generated by the pallet
- Nonce is marked consumed atomically at the moment the user submits their signed contract — same execution as mint
- Pallet rejects any submission referencing an already-consumed nonce
- Pallet rejects any submission where the current block exceeds the contract's expiry block
- These two checks together — nonce consumption and block expiry — are the complete replay prevention mechanism, both required

**General Build Requirements**
- All storage operations that touch related state must be atomic in the same execution — no partial updates
- `on_initialize` handles TTL expiry flips and 30-day grace-period purge checks every block
- Block-bound nonces on all genesis contract offers — replay prevention is non-negotiable
- SCALE encoding mandatory for all thumbprint inputs — raw byte concatenation prohibited
- RPC API returns expiry status, `is_active`, and `compromised_at_block` explicitly on every cert resolution
- Off-chain workers consume contract events to keep reputation scoring current
- Each layer independently upgradeable — pallet does not need to know about reputation or advisory contracts
